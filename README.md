# openfoam-cfd-minilab

Small OpenFOAM CFD mini-lab for a traditional incompressible-flow workflow. The included case is a 2D laminar lid-driven cavity at Reynolds number 100.

## Current Status

Case setup is provided and validated by repository tests. In this Codex environment on 2026-06-14, `blockMesh`, `checkMesh`, and `icoFoam` were not available on `PATH`, so no OpenFOAM simulation was executed here. This repository does not include fabricated residual histories, sampled velocity profiles, or field figures.

Simulation-derived files such as `figures/cavity_residuals.png`, `results/centerline_u.csv`, `results/centerline_v.csv`, `figures/cavity_centerline_profiles.png`, and `figures/cavity_velocity_magnitude.png` should only appear after running the OpenFOAM workflow locally.

## Governing Equations

The case uses the incompressible laminar Navier-Stokes equations:

```text
div(U) = 0
dU/dt + div(U U) = -grad(p) + nu laplacian(U)
```

where `U` is velocity, `p` is kinematic pressure, and `nu` is kinematic viscosity.

## Physical Setup

- Geometry: 2D unit square cavity, `0 <= x <= 1`, `0 <= y <= 1`
- Effective 2D treatment: thin `z` direction with `empty` front/back patches
- Lid velocity: `U = (1, 0, 0)`
- Characteristic length: `L = 1`
- Kinematic viscosity: `nu = 0.01`
- Reynolds number: `Re = U L / nu = 100`
- Solver: `icoFoam`, suitable for transient incompressible laminar flow

The GitHub Actions workflow targets OpenFOAM Foundation v11 using the `openfoam/openfoam11-paraview510:11` Docker image. For that solver environment, the tracked viscosity file is `cases/lid_driven_cavity/constant/physicalProperties`, which sets `nu = 0.01`. The older `constant/transportProperties` file is retained for compatibility with OpenFOAM variants that still read that dictionary.

## Mesh

The default case uses a `40 x 40 x 1` structured mesh generated by `blockMesh`. A coarser `20 x 20 x 1` mesh is also included. The run script selects from:

- `cases/lid_driven_cavity/system/blockMeshDict.20x20`
- `cases/lid_driven_cavity/system/blockMeshDict.40x40`

and copies the selected dictionary to `cases/lid_driven_cavity/system/blockMeshDict` before `blockMesh`.

Generated mesh files under `cases/lid_driven_cavity/constant/polyMesh/` are intentionally ignored by git and can be regenerated.

## Boundary Conditions

Velocity:

- `top`: moving wall, `movingWallVelocity`, value `(1 0 0)`
- `bottom`, `left`, `right`: no-slip walls
- `frontAndBack`: `empty`

Pressure:

- Walls: `zeroGradient`
- `frontAndBack`: `empty`
- Pressure reference: `pRefCell 0`, `pRefValue 0` in `fvSolution`

## Reproduction Commands

Run from the repository root inside an OpenFOAM-enabled shell:

```bash
bash scripts/run_cavity.sh
```

Run the included mesh refinements explicitly:

```bash
MESH_RESOLUTION=20 bash scripts/run_cavity.sh
MESH_RESOLUTION=40 bash scripts/run_cavity.sh
```

The script runs:

```bash
cp system/blockMeshDict.${MESH_RESOLUTION:-40}x${MESH_RESOLUTION:-40} system/blockMeshDict
blockMesh
checkMesh
icoFoam
postProcess -func sample -latestTime
foamToVTK -latestTime -fields "(U)"   # optional, only if foamToVTK is available
python scripts/plot_residuals.py --log results/logs/icoFoam.log --output figures/cavity_residuals.png --csv results/residuals.csv
python scripts/postprocess_cavity.py --case cases/lid_driven_cavity --results results --figures figures
```

Clean generated OpenFOAM and post-processing outputs:

```bash
bash scripts/clean_case.sh
```

Run repository tests:

```bash
python -m pytest
```

## Cloud reproduction with GitHub Actions

Local OpenFOAM commands were unavailable in the Codex environment used to create this scaffold, so the repository initially contains case setup and scripts, not executed solver results.

The manual GitHub Actions workflow at `.github/workflows/reproduce.yml` runs the solver stage in the `openfoam/openfoam11-paraview510:11` OpenFOAM Docker image. This OpenFOAM 11 workflow uses `constant/physicalProperties` for the kinematic viscosity. It then runs Python plotting and output validation on the Actions host with `actions/setup-python`, so the Docker container only needs to provide the OpenFOAM tools.

To run it:

1. Push this repository to GitHub.
2. Open the repository's **Actions** tab.
3. Select **Reproduce OpenFOAM Lid-Driven Cavity**.
4. Choose **Run workflow**.
5. After the workflow finishes, download the artifact named `openfoam-cavity-results`.

The workflow distinguishes case setup from executed solver results:

- Case setup is the tracked OpenFOAM dictionaries and scripts in this repository.
- Executed solver results are the logs, sampled CSVs, and figures generated by the workflow after `blockMesh`, `checkMesh`, `icoFoam`, and post-processing run.

The workflow fails if required outputs are missing or empty. It uploads `results/` and `figures/` as the `openfoam-cavity-results` artifact for inspection.

## Output Files

When the solver and post-processing run successfully, expected small outputs are:

- `results/logs/blockMesh.log`
- `results/logs/checkMesh.log`
- `results/logs/icoFoam.log`
- `results/logs/postProcess_sample.log`
- `results/residuals.csv`
- `results/centerline_u.csv`
- `results/centerline_v.csv`
- `figures/cavity_residuals.png`
- `figures/cavity_centerline_profiles.png`

Optional field visualization:

- `figures/cavity_velocity_magnitude.png`

The optional velocity-magnitude figure requires `foamToVTK` output or local ParaView/VTK tooling. If `foamToVTK` is unavailable, open the case in ParaView locally after running the solver.

## Limitations

- No simulation results are included from this environment because OpenFOAM was unavailable.
- The default mesh is a modest `40 x 40` teaching mesh, not a grid-convergence study.
- The centerline profiles are generated only from OpenFOAM sample files created by `postProcess`.
- The optional field visualization depends on VTK output; ParaView remains the recommended local inspection tool.
